# Approval Dashboard - Frontend Implementation

## Overview

Built a comprehensive React-based Approval Dashboard for reviewing and approving shopping lists generated by the Inventory Manager Agent. The dashboard provides a user-friendly interface for batch approval workflows.

## Files Created

### 1. TypeScript Types (`/frontend/src/types/approvals.ts`)
**Purpose**: Type definitions for shopping lists and approval workflow

**Key Types**:
- `ShoppingList`: Complete shopping list document
- `ShoppingListItem`: Individual item with forecasting context
- `UrgencySummary`: Breakdown of URGENT/STANDARD/LOW_PRIORITY items
- `SupplierBreakdown`: Supplier grouping for batch ordering
- `ApprovalRequest`, `PartialApprovalRequest`: Request bodies
- API response types for all endpoints

### 2. API Service (`/frontend/src/services/approvals.ts`)
**Purpose**: API client functions for all 7 approval endpoints

**Functions**:
- `getPendingApprovals()` - Get pending shopping lists
- `getShoppingListDetails(listId)` - Get detailed shopping list
- `approveShoppingList(listId, request)` - Approve entire list
- `approveShoppingListItems(listId, request)` - Partial approval
- `rejectShoppingList(listId, request)` - Reject list
- `getApprovalHistory(filters)` - Approval audit log (paginated)
- `getApprovalStats()` - Approval statistics

### 3. Main Component (`/frontend/src/pages/ApprovalsPage.tsx`)
**Purpose**: Main dashboard component with three tabs

**Features**:

#### **Pending Tab**:
- **Shopping List Cards**: Grid view of pending lists
  - List ID, generated date, total cost
  - Urgency summary badges (URGENT, STANDARD, LOW_PRIORITY)
  - Item count and confidence score
  - Click to view details

- **List Details View**:
  - Agent reasoning display
  - Item selection controls (Select All, Deselect All)
  - Comprehensive items table with:
    - Material name, current stock, daily demand
    - Days until stockout (highlighted if critical)
    - Quantity to order, cost
    - Urgency badge and reason
    - Supplier info with lead time
  - Supplier breakdown cards
  - Notes input for approval decision
  - Three action buttons:
    - **Approve Entire List** - One-click full approval
    - **Approve Selected Items** - Partial approval
    - **Reject List** - Rejection with notes

#### **History Tab**:
- Approval audit log table
- Status filter (All, Approved, Rejected, Partially Approved, Pending)
- Pagination (20 items per page)
- Columns: List ID, Generated, Status, Items, Total Cost, Reviewed, Notes
- Sortable and filterable

#### **Statistics Tab**:
- Overview cards: Pending, Approved, Rejected, Partially Approved, Total
- Approval breakdown chart with percentage bars
- Visual progress tracking

**State Management**:
- React hooks for tab state, pending lists, selected items
- Loading and error states
- Notes input for approval decisions
- Confirmation modal integration

## User Workflow

### 1. Daily Morning Routine (6:00 AM - 7:00 AM)

1. **Inventory Agent runs at 6:00 AM** (scheduled by orchestrator)
   - Analyzes current stock levels
   - Consults demand forecasts
   - Generates shopping list with urgency classification

2. **Admin logs in and navigates to Approvals tab**
   - Sees pending shopping list card with urgency summary
   - Example: "5 URGENT, 8 STANDARD, 3 LOW_PRIORITY items"

3. **Admin clicks list to view details**
   - Reviews agent reasoning
   - Sees items table sorted by urgency
   - Notes URGENT items (stockout <2 days)
   - Reviews supplier breakdown

4. **Admin makes approval decision**:
   - **Option A: Approve Entire List**
     - One-click approval
     - Optional notes: "Approved for ordering"
     - List status → `approved`

   - **Option B: Partial Approval** (e.g., budget constraints)
     - Select specific items (checkboxes)
     - Click "Approve Selected Items"
     - Notes: "Approved urgent items only, will review others Friday"
     - List status → `partially_approved`

   - **Option C: Reject List** (e.g., forecast seems wrong)
     - Click "Reject List"
     - Notes: "Please review demand forecasts, seems high"
     - List status → `rejected`

5. **Admin manually contacts suppliers** (not automated in MVP)
   - Uses supplier breakdown to batch orders
   - Example: "Fresh Meats Co: 3 items, ₹250.00"
   - Places orders via phone/email/portal

### 2. Weekly Sunday Routine (for LOW_PRIORITY items)

- LOW_PRIORITY items (stockout >7 days) accumulate
- Sunday morning: Review weekly digest
- Approve non-urgent items in batch

## API Endpoints Used

All endpoints under `/api/v1/approvals/`:

| Method | Endpoint | Purpose | Auth |
|--------|----------|---------|------|
| GET | `/pending` | Get pending shopping lists | Admin |
| GET | `/{list_id}` | Get detailed shopping list | Admin |
| POST | `/{list_id}/approve` | Approve entire list | Admin |
| POST | `/{list_id}/approve-items` | Approve specific items | Admin |
| POST | `/{list_id}/reject` | Reject list | Admin |
| GET | `/history?page=1&limit=20&status=approved` | Approval audit log | Admin |
| GET | `/stats` | Approval statistics | Admin |

**Authentication**: Session-based (HTTP-only cookies)
**Response Format**: Standard `success_response()` / `error_response()` helpers

## Technical Implementation

### React Patterns
- **Functional Components**: Modern React with hooks
- **useState**: Tab state, selected items, loading/error
- **useEffect**: Data fetching on mount and tab change
- **useMemo** (in HomePage): Tab filtering by role
- **Conditional Rendering**: Empty states, loading spinners

### State Management
- Local component state (no Redux needed)
- Optimistic UI updates after actions
- Error boundary for failed requests

### API Integration
- Axios client with interceptors
- `withCredentials: true` for session auth
- Error handling with try/catch
- Loading states during async operations

### Styling Approach
- CSS Modules (scoped to component)
- CSS variables for theming
- Flexbox and Grid for layouts
- Media queries for responsive design

## Performance Considerations

### Optimization
- **Pagination**: Limit history to 20 items per page
- **Lazy Loading**: Only fetch data when tab is active
- **Memoization**: Tab filtering memoized in HomePage
- **CSS Transitions**: Hardware-accelerated transforms

### Data Fetching
- Fetch pending lists only once per session
- Reload after approval/rejection actions
- No polling (use manual refresh button if needed)

```

  → User reviews and approves
  → Backend updates status → approved
  → Frontend reloads pending lists
```
